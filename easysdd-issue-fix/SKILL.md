# easysdd-issue-fix

本技能对应 easysdd-issue 工作流的**阶段三:修复验证**。

## 涉及的路径

> 路径约定见根技能 `easysdd` 第二节（组织规则 11）；issue 目录安排见 `easysdd-issue` 第二节。

## 你的职责

按确认的根因和方案**定点修复**,**不是重构,不是顺手优化**。修完之后验证,把修复记录写进 fix-note.md。这是 issue 工作流的收尾动作——没有验证闭环 = 工作流未完成。

## 两种入口

### 标准路径入口(有 analysis.md)

1. **根因分析存在且用户已确认方案**。读 issue 目录下的 `analysis.md`,确认 YAML frontmatter 有 `doc_type=issue-analysis` 且 `status=confirmed`,第 5 节"修复方案"用户选定了哪个方案。
2. **读全上下文**:
   - 根因分析(`analysis.md`)全文
   - 问题报告(`report.md`)全文
   - 根因分析第 1 节定位到的所有代码文件
   - `AGENTS.md`
   - 技巧库目录(`tricks/`)里与本 issue 相关的记录(可用 `python easysdd/tools/search-yaml.py --dir easysdd/tricks --filter status=active --query "{issue 关键词}"`),确认修复方式不违背已有库用法/模式建议
   - 探索归档目录(`explores/`)里与本 issue 相关的记录(可用 `python easysdd/tools/search-yaml.py --dir easysdd/explores --query "{issue 关键词}"`),确认修复点与已有证据不冲突
3. **确认修复起点**:告诉用户"我将按方案 X 修改 {文件列表},开始修复"——等用户确认才动手。

### 快速通道入口(无 analysis.md,从 easysdd-issue-report 直接触发)

进入此入口时,AI 在 report 阶段已读过代码并对根因有把握。

1. **向用户明确陈述根因**:"`{文件}:{行号}` 的 {具体代码} 存在 {问题描述}",让用户确认根因判断准确。
2. **给出修复方案**:改哪里、怎么改(一两句话即可,不写成完整分析文档)。
3. **等用户明确说"对,就这样改"后才动手**——不允许"我觉得对,直接改了"。
4. **读 `AGENTS.md`**。
5. **补搜技巧库目录(`tricks/`)**。快速通道也要用 `search-yaml.py` 查一遍同类技巧,避免采用已知反模式。
6. **补搜探索归档目录(`explores/`)**。快速通道也要用 `search-yaml.py` 查一遍同类历史探索,避免误把已知边界条件当成新问题。

## 强制规则

### 规则 1:只改根因分析里声明的文件

- 修复范围来自 analysis.md 第 5 节"推荐方案"里的"影响面"
- 超出范围的文件——哪怕顺眼——**不动**
- 发现范围外值得改的地方:记一条"顺手发现",不改代码

记录格式:
```markdown
> 顺手发现:{文件:行号} {问题简述}。不在本次修复范围,可后续另开 issue。
```

### 规则 2:改动最小化

- 修复的改动应该只针对根因,**不引入新的抽象、新的接口、新的模式**
- 如果发现"要把这个改好,需要先重构 X"——**停下来**,和用户确认是否要在这个 issue 里做重构,还是拆成独立的工作

### 规则 3:每完成一处改动必须汇报

汇报使用下面固定模板,**不允许含糊汇报**:

```markdown
## 修复汇报

### 动了哪些文件
{git status 输出或文件列表}

### 具体改了什么
- `{文件}:{行号}` — {改动描述,原来是 X,现在是 Y}
- ...

### 是否触碰到分析范围外的文件?
{是 / 否。是的话说明为什么}

### 是否引入了分析中没有提到的新概念/新结构?
{是 / 否。是的话停下来更新 analysis.md 后再继续}

### 复现步骤走一遍
{按 report.md 第 2 节复现步骤走一遍,结果是否符合期望行为?}
```

汇报完后**停下来**等用户回复。

### 规则 4:不做范围外的改动

和 easysdd-feature-implement 一样的逻辑:修复 issue 时发现的顺手优化机会,记录成新 issue,不顺手改。

## 验证清单

修复代码改完后,在汇报里逐项核对:

- [ ] **复现步骤验证**:按 report.md 第 2 节的步骤走一遍,问题不再出现
- [ ] **期望行为验证**:report.md 第 3 节描述的"期望行为"现在确实发生
- [ ] **影响面回归**:analysis.md 第 4 节"潜在受害模块"——每个模块走一遍最基本的冒烟路径,确认没被修复引入新问题
- [ ] **前端改动浏览器验证**(如果涉及):AGENTS.md 硬要求,不能只靠 typecheck
- [ ] **相关测试通过**:如果有测试覆盖到修复区域,跑一遍

## 修复未生效时:日志调试升级

走完验证清单后,如果发现**问题仍然复现**或行为与期望不符,不要在原有猜测上反复试错——切换到日志调试模式,重新收集运行时证据。

### 步骤

1. **宣告当前方案未解决问题**。告诉用户:"按方案 X 的改动未能消除问题,准备通过日志进一步定位。"
2. **确定打点位置**。根据 analysis.md 第 1 节定位到的代码路径,在关键分支、疑似错误路径、数据流入口/出口打上日志(log 语句 / print / 结构化 logger,视项目技术栈而定),记录关键变量的值和执行路径。
3. **获取日志**:
   - **如果 AI 可以自己执行代码并拿到输出**(有终端权限、测试可以跑、本地服务可以启动),直接运行复现步骤并把日志输出粘贴到工作区,自行分析。
   - **如果 AI 无法自行执行**(需要用户操作客户端、特定环境、无法自动化的 UI 路径),明确告诉用户:
     ```
     请按以下步骤复现问题并粘贴日志:
     1. {具体复现操作}
     2. 复制控制台/日志文件里从 [打点标记 A] 到 [打点标记 B] 的输出
     ```
4. **分析日志,修订根因假设**。拿到日志后:
   - 对照 analysis.md 第 2 节"根因假设",判断原假设是否成立
   - 如果原假设被证伪,**回填 analysis.md**:在第 6 节(或新增一个"调试补充"小节)记录新证据和修订后的假设,不要悄悄改第 2 节的原始假设
5. **清理日志打点**。拿到结论后立即移除所有临时 log 语句,不允许调试日志进入最终提交。
6. **以修订后的根因重新进入修复流程**。

### 循环限制

- 如果经过 **2 轮日志调试**后仍未定位到根因,停下来,主动告知用户:"当前信息不足以通过日志定位,建议重新走 `easysdd-issue-analyze` 更新根因分析。"不要无限循环。

## 写 fix-note.md

验证通过后,在 issue 目录下创建 `fix-note.md`(位置见主技能 `easysdd-issue` 第二节),记录本次修复的完整闭环。

**标准路径模板**(有 report.md + analysis.md):

```markdown
---
doc_type: issue-fix
issue: {issue 目录名}
path: standard
fix_date: YYYY-MM-DD
related: [{analysis.md 相对路径}]
tags: []
---

# {问题简述} 修复记录

## 1. 实际采用方案

{方案 A / B,或说明与 analysis.md 确认方案的偏差及原因}

## 2. 改动文件清单

- `{文件}:{行号范围}` — {改动说明}
- ...

## 3. 验证结果

- 复现步骤验证:{通过 ✓}
- 期望行为验证:{通过 ✓}
- 影响面回归:{通过 ✓ / 发现问题 X(处理方式)}

## 4. 遗留事项

- {顺手发现的后续 issue(如有)}
- {已知限制(如有)}
- 无
```

**快速通道模板**(无 report.md / analysis.md):

```markdown
---
doc_type: issue-fix
issue: {issue 目录名}
path: fast-track
fix_date: YYYY-MM-DD
tags: []
---

# {问题简述} 修复记录

## 1. 问题描述

{一句话说清楚发生了什么,在哪里发现的}

## 2. 根因

`{文件}:{行号}` — {为什么这里有问题,一两句话说清楚}

## 3. 修复方案

{改了什么、改动说明}

## 4. 改动文件清单

- `{文件}:{行号范围}` — {改动说明}
- ...

## 5. 验证结果

{用户如何确认修复有效,结果如何}

## 6. 遗留事项

- {顺手发现的后续 issue(如有)}
- 无
```

## 退出条件

- [ ] 所有改动文件已提交或列清单
- [ ] 验证清单全部勾选
- [ ] fix-note.md 已创建并填写完整(标准路径用标准模板,快速通道用快速通道模板)
- [ ] 没有未处理的"顺手发现"(都进了后续 issue 列表)
- [ ] 没有范围外的改动(或已和用户确认)
- [ ] 用户明确确认修复完成

## 收尾提交

按根技能 `easysdd` 第五节约束 9"收尾提交（scoped-commit）"的规则执行。本阶段的特定要点：

- **提交范围**：修复代码、`fix-note.md`，以及本次确实一并更新的 `report.md` / `analysis.md`
- 修复闭环后先告诉用户"修复验证已完成，`fix-note.md` 已落盘"，紧接着问是否需要 commit

## 退出后

先告诉用户:"issue 修复完成,工作流闭环。问题报告(report.md) + 根因分析(analysis.md) + 修复记录(fix-note.md)已存档。"

然后**补问一次**是否需要你代为提交本次修复;用户同意时,按收尾提交规则执行到 commit 完成。

可以建议:
- 把 issue 目录下的文件和代码改动放在同一次提交里,方便日后追溯
- "顺手发现"的后续 issue,另开一轮 `easysdd-issue-report` 处理,不要在这个 PR 里塞

如果修复过程中发现问题实际上是功能缺失(不是 BUG),建议用户另开 `easysdd-feature` 工作流处理,不要在 issue 工作流里偷偷做新功能。

## 反模式(看到就停)

- ❌ 修完代码没走验证清单就宣告"修好了"
- ❌ 顺手改了根因分析范围外的代码
- ❌ 修复引入了新的抽象/接口但没停下来和用户确认
- ❌ fix-note.md 没有创建就宣告工作流完成
- ❌ 发现影响面回归有问题,但在汇报里写"轻微影响,可以忽略"——不行,要修到干净
- ❌ 前端有改动但没在浏览器跑,只 typecheck 就报通过
- ❌ 用户没明确说"修复完成"就结束工作流
- ❌ 修复未生效但继续在原假设上反复猜测试错,不切换到日志调试
- ❌ 日志调试结束后没有清理临时 log 语句就提交代码
- ❌ 工作流收尾时没有询问用户是否需要代为 commit 本次修复
- ❌ 用户没明确同意就直接 `git commit`

# 编译器驱动重构：为什么不靠测试防护网

这份文档是三张处方背后的共同哲学。SKILL.md 里的"四条纪律"和"三步循环"都来自这里。

---

## 悖论：没测试不敢重构，要重构才能写测试

经典敏捷教义说"重构之前先建测试防护网"。理论没错，但在真实遗留代码里会死在第一步：

- 模块耦合太紧，单测要 mock 十几个依赖
- 副作用藏在层层调用里，测试要把副作用都 stub 掉
- 业务分支多到列不全，测试覆盖率永远不够
- **最要命的是：为了让代码"可测"，你不得不先重构——但你本来就是为了"能重构"才写测试的**

这就是 Arlo Belshee 点出的"test-first 悖论"。他的答案：**换一种防护**。

---

## Arlo 的 Provable Refactorings

核心主张：如果一个重构操作能让**编译器 / 静态分析器**证明"行为等价"，那就不需要测试来二次证明。风险模型变成"只有编译器本身有 bug，这个重构才可能引入 bug"——这个概率比"测试没覆盖到的分支"低几个数量级。

什么样的操作能被编译器证明？三类：

1. **纯搬运**：把一个字段 / 方法 / 类型从 A 移到 B，引用全部跟着改。IDE 的 Move 重构做这个。
2. **纯重命名**：符号改名，所有引用同步改。IDE 的 Rename。
3. **纯抽取**：把一段代码抽成函数/变量，调用点替换为调用新符号。IDE 的 Extract Method / Extract Variable。

反过来，不能被编译器证明的（需要人脑或测试）：
- 改条件判断
- 改循环边界
- 调用顺序换
- 算法替换
- 类型契约收紧/放宽

**结论**：只用第一类操作，就能在零测试的前提下把代码"搬"成完全不同的结构。行为等价是编译器给的保证，不是人承诺的。

## The Core 6（Arlo 给的高负债代码常用 6 招）

这 6 招是"永远安全"的，多语言都有 IDE 支持：

| 招 | 做什么 | Vue 场景举例 |
|---|---|---|
| Rename | 符号改名 | `data` → `userProfile` |
| Inline | 把函数体展开到调用点 | 只被调一次的私有方法 |
| Extract Method | 抽一块代码成函数 | 深嵌套 if 里的一块 |
| Extract Variable | 抽一个表达式成变量 | 复杂 computed 的中间结果 |
| Move | 把符号从 A 搬到 B | 一个 ref 从组件搬到 composable |
| Introduce Parameter | 把局部变量变成函数参数 | 为了让函数能被外部调用 |

外加 3 个"工具函数式"操作：Inline Variable、Change Signature（只加参数不删）、Safe Delete（检查无引用再删）。

全部加起来不到 10 种操作，**能覆盖 80% 的遗留代码改造**。

## "Lean on the Compiler"（Michael Feathers）

比 Arlo 早 10 年，Feathers 在 *Working Effectively with Legacy Code* 里说的是同一件事的另一面：

> 与其自己在脑子里推理"如果我改了这个，哪里会断？"，不如**故意做一个会让编译器报错的改动**，让编译器替你列出"哪里会断"的完整清单。

三步法：
1. 做出一个你知道会让编译器不高兴的改动（比如把一个方法签名改了，或者把一个字段改成 private）
2. **读编译错当作调用点清单**——这是免费的、完备的、零遗漏的
3. 按清单一个个修

这个直觉非常反常识。大多数开发者看到红色错误提示本能地想"糟糕、赶紧消掉"。Feathers 的思路是"太棒了、清单来了"。

在 Vue + TypeScript 场景里，这个杠杆特别好用——Volar 会精确告诉你：
- 哪个 `.vue` 文件的 template 里引用了已被改名的变量
- 哪个 composable 的 return 签名变了、哪些调用方在解构它
- 哪个 `defineProps` 的类型和父组件传的不一致

## 三步循环：创建新 → 替换引用 → 删除旧

Nicolas Carlo 把这个节奏叫 "6 Safe Refactorings for Untested Legacy Code"，核心是三步：

```
┌─────────────────────────────────────────────┐
│  Step 1: 在新位置创建目标符号（空壳也可以）  │
│  → 编译仍然绿：新的还没被用到               │
├─────────────────────────────────────────────┤
│  Step 2: 逐个把调用点从旧符号改到新符号      │
│  → 每改一个编译都应该绿                     │
├─────────────────────────────────────────────┤
│  Step 3: 旧符号无引用时，Safe Delete 删掉   │
│  → 编译仍然绿：没人在用了                   │
└─────────────────────────────────────────────┘
```

关键点：**Step 2 的每一小步都是独立可提交、可回滚的**。如果搬了一半发现路径选错了，回滚只丢这一小步的工作，不会连累前后。

对比"直接 Rename Symbol"：IDE 会一次性把 100 个引用全改掉。看起来效率高，但：
- 万一改错了，整个一次性回滚（或手动逐个改回来）
- 中间没有"编译错当清单"的机会——编译一直是绿的，但这种绿只证明"引用指向对了"，不证明"你打算搬的真是这些地方"
- 无法分两次会话做

所以 **Rename Symbol 其实是错误的默认选择**。真正安全的是 Move + 手动逐点替换。

## 在 Vue 里这个方法论如何具体落地

Vue 有三个 TS 不太管得住的地方，要额外小心：

1. **Template 里的引用**：`<template>` 里用的变量靠 Volar 的 template 类型推断。改 `<script setup>` 里的符号时，template 不一定报错。应对：每次搬完在 VSCode 打开对应 `.vue` 文件肉眼扫一眼 template，或者跑 `vue-tsc --noEmit`。
2. **Options API 的 `this`**：`methods` / `computed` / `data` 之间通过 `this` 互相访问，TS 能推但推得不如 setup 好。在 Options API 项目里做重构，优先级是"先把一簇相关的东西搬出去"，而不是"跨 data/methods 内部重排"。
3. **Pinia store 的 `storeToRefs`**：解构 store 时 reactive 语义很微妙。重构涉及 store 访问点时要留意：搬运过程中不要顺手把 `store.count` 改成 `storeToRefs(store).count`——那是行为改变（响应式语义变了）。

## 什么时候仍然需要测试

编译器驱动重构**不是**"永远不写测试"。它是说"重构本身不需要测试来保证安全"。测试在以下场景仍然不可替代：

- **行为验证**：重构完跑一轮回归测试确认"外部可观察行为真的没变"。这是最后一道网，不是重构过程中的网。
- **行为变更**：要改行为时（feature / bugfix），测试是必需的保护。
- **纯业务逻辑锁定**：处方 C 把纯函数抽出来后，这些纯函数很适合补单测——因为它们现在真的**能测**了。重构先、补测后，顺序不能反。
- **UI 回归**：前端要目视验证，编译器看不见布局错位。关键页面重构完要肉眼跑一圈。

---

## 一句话总结

> 重构不是"把代码改好"，是"把代码里的信息量逐步暴露出来"。编译器是帮你暴露信息的 X 光机——不用它是吃亏。

这句话也是用户老师说的"水落石出"的 Arlo 版本：水不是被戽干的，是被一勺勺舀到别的地方去的，舀的过程里石头（主干）自然显形。

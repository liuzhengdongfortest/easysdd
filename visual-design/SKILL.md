# visual-design

能让一个设计放十年还不过时的人——Dieter Rams、Jonathan Ive、Steve Jobs、Massimo Vignelli、Paul Rand、原研哉、深泽直人、Josef Müller-Brockmann、Edward Tufte、Don Norman——做设计时心里都装着同一句话：

**我的活不是在作品上加东西让它变漂亮，是在作品上拿掉东西让它本来的秩序和功能显露出来。每加一个元素前都要先问——它配占用这份注意力吗？**

这份技能有两种工作模式——

**没有设计稿**：用户描述了一个系统或功能，还没有 UI。这时候的工作是从需求出发，规划出 UX 应该长什么样——任务流、信息架构、交互模式、边缘状态——直到输出一份可以被评审的设计方案。

**已经有设计稿**：用户带来了已有的设计。这时候的工作是作为高级设计师走查这份稿子——先看结构（流程对不对、找得到吗、认知负担、反馈是否到位），再看视觉（减法、层次、对比、一致性）——输出具体的问题清单和改进建议，不是泛泛而谈。


## 从功能需求到 UX 设计

用户拿来一份功能列表，或者口头描述一个系统要做什么——这时候的工作是把功能需求翻译成 UX 设计。这不是画界面，是**在动手之前把结构想清楚**。

Apple 设计团队在做一个新产品时，最先讨论的从来不是"界面长什么样"，是"**这个人要完成什么，用最少的步骤**"。Jony Ive 多次讲过：他们通常会把一个想法砍掉 90% 的功能，只留下那个最核心的一件事，然后把那件事做到极致，再去想界面。

**第一步：把功能列表翻译成用户目标**

功能列表通常是从工程师或产品经理角度写的——"支持上传图片"、"可以添加评论"、"有通知功能"。这些不是用户目标，是实现手段。用户目标是这样的：

- "我要快速分享一张图给朋友"
- "我要知道我的项目有没有新动态"
- "我要找到上周某天的记录"

对每个功能问"**用户为什么要用这个**"——答案就是用户目标。先把功能列表翻译成用户目标，再来规划 UX。多个功能可能对应同一个用户目标，这时候它们在设计上应该在一起。

**第二步：规划主任务流**

确定最重要的一条路——用户完成核心目标的完整路径。这条路要尽量短。对路上的每一步问：**能不能去掉，或者合并进上一步？** 删不掉的才留下。

写成这样：

```
用户目标：[一句话]

主任务流：
[入口] → [步骤1] → [步骤2] → ... → [完成]

每步说明：
- 步骤1：用户看到什么 / 做什么 / 做完发生什么
- 步骤2：...
```

超过 5 步的流程几乎总是可以压缩的。Amazon 1-Click 是极端案例——从"我要买这个"到"交易完成"一步。不是所有产品都能做到一步，但每减一步都是在减放弃率。

**第三步：组织信息架构**

主任务流规划完之后，组织整个系统的结构——哪些功能并列、哪些从属、哪些通过不同入口到达。

信息架构要按**用户的任务逻辑**组织，不是按功能分类。"账号设置"里放"分享权限"——用户要找分享功能时就会找不到，因为 TA 不会想到去"账号"里找。应该放在离分享操作最近的地方。

写成缩进层级：

```
一级导航：A / B / C
  A 下面：A1 / A2
  B 下面：B1 / B2 / B3
```

每一级问：用户带着任务 X 来，第一眼会去哪找？答案和实际层级不一致，就调整。核心功能超过 3 层点击才能到的，考虑提层。

**第四步：处理边缘状态**

主流程想清楚之后，处理三种边缘状态——每种都容易被忽略，但每种都是用户体验的关键时刻：

- **空状态**：用户第一次使用、还没有数据时看到什么？不要只是空白——这是引导用户开始的机会，告诉 TA 下一步做什么
- **错误状态**：出错了告诉用户什么？错误信息要说清楚发生了什么、能怎么解决，不是 "出错了请重试"
- **加载状态**：有延迟时用户看到什么？进度要可见，不要让用户以为卡死了

**第五步：输出设计方案**

前四步想清楚之后，输出一份可以被评审的设计方案：

```
【设计方案：[系统/功能名称]】

用户核心目标：
[一句话]

主任务流：
Step 1 — [屏幕/状态名称]
  用户看到：...
  用户操作：...
  反馈/跳转：...

Step 2 — ...

信息架构：
[缩进层级]

关键设计决策：
- [做了什么选择，为什么这样做而不是另一种方式]
- ...

边缘状态：
- 空状态：...
- 错误：...
- 加载：...

待确认：
- [需要用户回答的问题：某个功能的优先级、某个场景是否存在、某个约束条件]
```

"待确认"部分不是推卸决策责任，是把真正需要产品/业务判断的问题单独列出来——设计师能决定的先决定，需要额外信息的明确标出来。


## UX 结构诊断

拿到一份已有的设计稿，先不看视觉。大部分界面上"看起来不对"的地方，真正的病根在结构，不在像素。视觉打磨放在结构诊断之后。

**先过这四关：**

**1. 任务流：用户能完成吗**

从入口走到完成，共几步？对每一步问：这步能合并或去掉吗？

Steve Krug 的一句话：**"每增加一个思考点，都在消耗用户不会退款的耐心。"** 流程里每一步都是放弃的机会。找出最多余的两步，提议合并或删掉，给出理由。

**2. 信息架构：用户找得到吗**

用户想做 X，按 TA 的逻辑去找，路径通吗？还是设计者按内部逻辑分类，导致 TA 需要到意想不到的地方才找得到？

"如果用户找不到它，它就不存在。" 找出分类维度和用户心智模型最不吻合的那一处，指出来，给出调整建议。

**3. 认知负担：用户被要求记住或决定多少**

- **记忆负担**：有没有让用户记住上一步的信息才能完成这一步的地方？如果有，把信息搬到同一屏
- **决策负担**：台面上同时呈现的操作选项超过 5 个吗？次要操作是否可以折叠

Miller's Law：用户短期记忆能处理的独立信息块约 7±2 个，在任务执行中能同时维持的注意力通常不超过 3-4 件。超过这个数量，用户开始犯错、放弃。

**4. 状态与反馈：用户知道发生了什么吗**

Nielsen 十条可用性原则第一条：**系统状态要可见（Visibility of System Status）**。

对每个有后果的操作（提交、删除、上传、支付）问：用户做完之后，界面给 TA 的反馈是什么、在哪里、多快出现？

反馈要做到：**及时**（动作完成后立刻，不是几秒后）、**在视线内**（错误提示在产生错误的输入框旁边，不是页面顶部）、**明确**（"邮箱格式不对"比"输入有误"好，"操作成功"比"200 OK"好）。

**输出诊断意见的格式：**

```
【UX 诊断：[设计名称]】

严重问题（用户会卡住或流失）：
- [具体问题描述] → 建议：[具体改法]

中等问题（用户会迷惑或绕路）：
- [具体问题描述] → 建议：[具体改法]

轻微问题（体验不顺滑但不影响完成）：
- [具体问题描述] → 建议：[具体改法]

视觉层问题（结构已经 OK，以下是视觉打磨）：
- ...
```

诊断意见要具体——不是"这里交互有点怪"，是"Step 2 的确认弹窗在用户刚填完表单时出现，但弹窗里没有显示用户填的内容，用户无法确认自己是否填对了，建议在弹窗里展示关键填写内容"。


## 先问"能不能不加"

Dieter Rams 有十条设计原则，最狠的一条是 **"Good design is as little design as possible"**——好设计就是尽量不做设计。

你在做 UI 时最常干的事是往页面上放东西——放一个按钮、一张卡片、一个图标、一条分隔线、一个 tooltip。每一个放下去当下都有理由（"这里加个分隔线读者才知道是两块"、"这里加个图标页面才不会太冷"）。但十个"有理由的加法"叠起来，页面就乱了。

大师的动作反过来——每次想加一个元素时先问"**不加会怎样**"。

- 这条分隔线不加会怎样？——其实两块内容之间的间距已经足够区分了。删掉。
- 这个图标不加会怎样？——文字已经讲清楚了，图标只是装饰。删掉。
- 这个背景色块不加会怎样？——页面会显得"空"，但空不是问题，空是留白。删掉。

具体动作：设计稿画完后，把每个元素挨个盖掉试一次——盖掉之后页面表达的信息少了吗？没少就删掉那个元素。


## 设计是让东西清楚，不是让东西好看

Steve Jobs：**"Design is not just what it looks like and feels like. Design is how it works."**

**好看是结果，不是目标。** 目标是让这东西的功能、层次、顺序一眼就被理解。理解对了，好看自然就在那里。目标搞反了，先追求好看，功能就被装饰盖住。

具体来说：

- 一个登录页的目标不是"好看"，是让用户 3 秒内看到输入框在哪、按钮在哪、如果忘了密码去哪
- 一张海报的目标不是"炫酷"，是让路人走过时在 1 秒内看到它讲的是什么活动、什么时间、什么地点
- 一个 dashboard 的目标不是"专业"，是让看的人一眼看到哪个指标出问题了

做完一个设计后不要问"好看吗"，问"**主要信息在 3 秒内被看到了吗**"。

具体动作：每做完一稿设计把自己当成第一次看这东西的读者，给自己 3 秒——3 秒内你看到的是什么？如果看到的是装饰、图标、背景而不是核心信息，调整元素的视觉权重（主信息更大更重，装饰更小更淡）直到 3 秒内第一眼落在核心信息上。


## 留白不是省出来的，是主角之一

原研哉是无印良品的艺术总监，他讲过一个词——**"空"（ku）**。留白不是没填满的地方，是跟文字和图形同等重要的设计元素——决定了读者的眼睛往哪看、在哪停、以什么速度读。

Apple 的页面留白极多——一张产品页上可能 70% 是空的。这不是懒，是精算——留白让剩下那 30% 的内容显得贵、显得值得细看。四周越空，中间那个 iPhone 就越有存在感。

具体动作：看自己的设计稿时问——"这上面的留白，是我主动留的，还是因为没东西放才剩下的？"如果是后者，说明你还没把留白当成设计元素。主动留白的方法：先按内容需要的量排好，然后把周围的空白再加一倍。


## 字体选少，层次靠字号和字重

Massimo Vignelli：**"在我职业生涯里，我用过的字体不超过 5-6 种。一个设计师用 200 种字体，只是证明 TA 不懂设计。"**

层次不是靠字体种类拉开的，是靠字号、字重、颜色拉开的。一个干净的设计往往只用一种字体家族，但在字号上有 3-4 级（比如 12 / 16 / 24 / 48），字重上有 2-3 档（比如 Regular / Medium / Bold）。Apple 的 SF Pro 就一种，靠 9 种字重 + 多种字号完成所有 iOS 界面的层级。

具体动作：字体家族最多 2 个。字号用固定梯度（比如 12 / 14 / 16 / 20 / 24 / 32 / 48），不要随手输入任意数字。字重最多 3 档。超过就是在用字体做装饰，不是在做层次。


## 对齐不是美学，是秩序

Josef Müller-Brockmann 一辈子在讲一件事——**栅格系统**。所有元素落在看不见的网格上，上下左右严格对齐。

对齐不是洁癖，是秩序感的来源。读者的眼睛能快速扫过一个对齐的版面，因为有路可走；读者的眼睛会在不对齐的版面上反复跳跃，因为每一行都要重新寻找起点。

Apple 所有界面都有严格的 8pt 栅格——任何元素的大小、位置、间距都是 8 的倍数。

具体动作：
- 做界面设计先定栅格（常见 4pt 或 8pt 基准），所有元素的宽高间距都是基准的整数倍
- 同一层级的元素必须左边对齐到同一条线
- 同一区域里只用一种对齐方式（左/右/居中），混着用就显得乱


## 颜色克制，灰度先行

Dieter Rams 博朗设计的作品大部分是黑、白、灰——偶尔一个红色按钮强调一个最重要的动作。**颜色是最贵的注意力资源，一花就没了。**

你在界面上用了 5 种鲜艳颜色，就等于对读者喊了 5 句"看我"——TA 不知道该先看哪个。只用一种鲜艳颜色、其他全部灰度，那一抹颜色就指向了你最想让 TA 看的东西。

大师的顺序反直觉——**先用纯灰度做一遍**，把所有的层次、对比、主次都在灰度版本上调好，再加一种主色（品牌色）用在最关键的动作或状态上。

具体动作：
- 先用灰度设计，直到布局和层次都立住了再加颜色
- 每个界面主色不超过 1 种，其他全部灰度
- 红、绿、黄这类有强烈语义的颜色只用在语义对的地方（红=危险/删除，绿=成功，黄=警告），不当装饰色


## 对比要狠，不要温

Edward Tufte：**让最重要的东西最显眼。** 听起来像废话，但大部分人做的时候把所有东西都弄得差不多重要，结果没有一个被注意到。

新手经常做"温和的对比"——标题比正文大一号、加粗一档——温和到读者感受不到层次差。大师做对比会故意做**狠**——标题比正文大 3-4 倍、正文用浅灰、caption 用更浅的灰——读者一眼看到主次。

Apple 发布会 Keynote 是这个原则的教科书——一张 slide 上只有一个巨大的词（比如 "One more thing."），其他全空。对比极致，信息就像从屏幕里砸出来一样。

具体动作：设计完后眯着眼看稿子——细节模糊后只剩明暗块。如果看到一片差不多深浅的灰 = 对比不够狠；一眼能看到"最重要的那个深色块" = 对比立住了。


## 动效是指路，不是炫技

Jonathan Ive：**"Motion should be purposeful, not decorative."**

动效在告诉读者"刚才发生了什么"——动作被接收了、结果在哪里出现、新页面从哪个方向来。这是动效的语言功能。为了炫技的动效（3 秒加载动画、弹跳、视差）第一次看觉得惊艳，第二次就觉得慢，第三次就想关掉。

好动效的特征：短（200-300ms，超过 400ms 读者就开始等）、有方向（从右滑入 = 进入更深层，从上落下 = 这是覆盖层）、有缓动（自然的加速/减速曲线）、**第二次就应该让读者忽略**。

具体动作：每个动效做完后问——"这个动效在告诉读者什么信息？"答不出来的删掉。


## 一致性比创意更重要

Don Norman 在《设计心理学》里讲 **affordance（可供性）**——一个东西的外观暗示了它该怎么用：门把手暗示"拉"，按钮的凸起暗示"按"，链接的蓝色下划线暗示"点击"。affordance 是靠一致性维持的。

如果同一个动作（删除）有时候叫"删除"、有时候叫"移除"、有时候只是一个 X 图标；同样的元素（比如警告）有时候红色、有时候橙色——读者失去了学习的能力，每次都要重新判断。

Apple 的 HIG 几百页都在讲同一件事——同样的场景用同样的控件，同样的动作用同样的词，同样的反馈用同样的形式。

具体动作：
- 同一个产品里按钮样式不超过 3 种（主/次/文字按钮），每种只对应一类功能
- 同一个动作全站只用一个词
- 建一份设计系统文档——颜色、字号、间距、控件样式固定下来，之后做新页面从库里拿


## 学大师先抄理由，再谈风格

新手看到好看的设计稿，本能是抄外观——一样的颜色、字体、阴影、圆角。抄完发现自己的作品还是不好看——因为只抄了结果，没抄到产生这个结果的理由。

- Apple 用大面积留白——不是因为留白好看，是因为 Apple 想让产品显得"贵"。如果你的产品是急着吸引点击的电商促销页，大面积留白只会让转化率下降。
- Material 用阴影和浮层——不是因为阴影好看，是因为 Material 要在 Android 上解决层级识别问题。如果你在做 Web 端企业后台，阴影只会让界面显得又重又慢。

具体动作：看到喜欢的设计，先问三个问题——这个设计的用户是谁？它在传递什么情绪？它优先解决什么问题？如果三个答案跟你自己项目的答案不一致，抄它的外观就会翻车。


## 怎么判断一个设计好不好

**任务测试**：找一个没用过这个产品的人，给 TA 一个任务（"假设你要完成 X，操作一下"），然后闭嘴观察。TA 在哪里犹豫、走错、放弃——哪里卡，哪里就是问题。这是所有测试里最有效的一个，哪怕只找一个人，看 TA 操作 5 分钟，暴露的问题会比你自审一小时多得多。

**眯眼测试**：眯起眼睛看设计稿，细节模糊，只剩明暗块。层次清不清楚、主次有没有、对比够不够——一眼看出来。

**灰度测试**：把稿子变成黑白灰，看它还 work 不 work。灰度版本就已经清楚了，加颜色只是锦上添花；灰度版本里层次全垮，说明你的设计在靠颜色撑场面。

**拿掉一半测试**：强制从稿子里删掉一半元素，看剩下的功能还在不在。大部分时候功能没受影响，删掉的那一半就是多余的。

**3 秒测试**：给自己 3 秒扫一眼稿子，闭眼回忆第一眼看到的是什么。如果是你想让读者看到的主信息，对了；如果是背景装饰、某个 banner、某个图标，视觉权重错位了。


## 写完扫一眼

**结构层**
- 主任务从进入到完成共几步？超过 5 步的找出最多余的两步提议合并
- 用户按自己的逻辑找核心功能，路径通吗？核心功能超过 3 层点击才能到的，考虑提层
- 台面上同时呈现的操作选项超过 5 个了吗？有没有让用户跨屏记住信息的地方？
- 每个有后果的操作（提交、删除、上传、支付），完成后反馈是什么、在哪里、多快出现？
- 错误提示是在产生错误的地方旁边，还是在页面别的地方？
- 空状态、错误状态、加载状态都处理了吗？

**视觉层**
- 每个元素都问过"不加会怎样"了吗？盖掉不影响表达的删掉
- 3 秒扫一眼，第一眼落在主信息上还是装饰上？
- 留白是主动留的还是剩下的？
- 字体家族超过 2 个？字号是固定梯度还是随手输的？字重超过 3 档？
- 所有元素落在同一栅格上吗？同层级元素左边对齐到同一条线吗？
- 主色超过 1 种？灰度版本能单独讲清楚层次吗？
- 眯起眼看，最重要的那个元素是最显眼的那个块吗？
- 动效答得出在说什么信息吗？答不出的删掉
- 同样的动作在全站用了同样的词和样式吗？

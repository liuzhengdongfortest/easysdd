# vue-architecture-optimizer-framework

把 Vue 架构优化做成可复用流程，而不是一次性的“重构冲动”。

这个 skill 解决三类任务：
1. 已有代码 + 业务功能将变化：先理解业务目标，再做结构优化。
2. 已有代码 + 纯技术问题：可直接做技术重构（如抽 composable、拆层、统一状态流）。
3. 暂无代码 + 需要先设计架构：基于业务流程先出可落地的 Vue 架构蓝图。

## 核心心智模型

### 1. 业务边界先于技术边界
技术分层不是目的，业务变化成本才是目的函数。先找业务能力边界，再映射到模块边界。

### 2. 稳定内核与高变外层分离
把稳定规则放在内核层，把高频变化放在外层编排层，降低连锁改动。

### 3. 数据流必须可追踪
同一份关键状态必须有单一事实来源，流向要能被开发者在 30 秒内说清。

### 4. 复用基于语义而非文本
不是“重复代码就抽”，而是“重复意图才抽”。抽象单位应对应业务语义。

### 5. 渐进式重构优先于大爆炸重写
默认以可回滚的小步重构推进，只有在收益显著大于迁移风险时才建议重写。

## 决策启发式

1. 如果改动依赖业务规则变化，先产出业务流程图或规则清单，再动结构。
2. 如果问题主要是重复状态逻辑，优先抽 composable，不先抽 UI 组件。
3. 如果一个模块同时处理“取数 + 业务计算 + 视图状态”，先拆职责后谈性能。
4. 如果接口层不稳定，先隔离 adapter/bff 映射层，避免污染页面层。
5. 如果状态跨 3 个以上页面共享，优先放到 Pinia store 或领域 store，而非 props 透传。
6. 如果改动无法做到增量发布，默认降级方案复杂度，拆成两轮。
7. 如果命名无法体现业务意图，架构还没设计好，不进入实现。
8. 如果收益无法量化（复杂度、耦合度、交付速度），不做大范围结构调整。

- “优化 Vue 架构”
- “这段 Vue 代码结构太乱了”
- “帮我重构成 composable”
- “页面越来越难维护，怎么拆层”

## 不适用范围

- 纯视觉设计决策（交给 UX/视觉类 skill）
- 无业务约束的“为了优雅而重构”
- 一次性脚本页面，不需要长期维护的原型

## 回答工作流（Agentic Protocol）

核心原则：先判定问题类型，再选择优化路径；业务相关改动不盲改，技术债可快改。

### Step 1: 问题分类

| 类型 | 识别特征 | 行动 |
|---|---|---|
| 业务驱动优化（已有代码） | 需求包含新功能/业务规则变化 | 先澄清业务目标和约束，再做结构方案 |
| 技术驱动优化（已有代码） | 需求是复杂度高、重复逻辑、状态混乱 | 直接做代码结构诊断和增量重构方案 |
| 架构先行设计（无代码） | 还未实现，需要先定义工程架构 | 基于业务流程设计分层、目录、状态和演进路径 |

判断原则：只要结构调整会影响业务行为，就先确认业务语义；不会影响业务语义的纯技术债可直接优化。

### Step 2: Vue 架构研究（必须基于事实）

根据心智模型，按以下维度收集证据后再给出方案：

#### 维度 A：业务边界与变化方向
- 核心业务流程有哪些步骤，哪些规则最可能变化。
- 页面/模块与业务能力是否一一对应。
- 哪些逻辑是“业务计算”，哪些只是“UI 编排”。
- 未来 1-2 个迭代最可能新增的能力点。

#### 维度 B：模块职责与耦合图
- 页面、组件、composable、store、service 的职责是否混叠。
- 是否存在循环依赖、隐式共享状态、跨层调用。
- 关键功能变更需要改多少文件、跨几层。
- 目录结构是否反映领域语义而非技术堆叠。

#### 维度 C：状态流与副作用管理
- 状态源是否唯一，是否存在镜像状态和双向同步。
- 异步请求、缓存、错误处理、重试策略分散在哪些层。
- 副作用是否集中在可测试边界（store/composable/service）。
- 是否能快速定位“谁在改这份状态”。

#### 维度 D：复用与抽象质量
- 重复逻辑是否是同一语义，还是偶然相似。
- composable 是否承载业务能力，而非杂糅工具函数。
- 组件拆分是否围绕“可替换性”和“可测试性”。
- 是否出现过早抽象或抽象泄漏。

#### 维度 E：迁移风险与收益
- 能否按小步提交和灰度上线。
- 是否有回滚路径和兼容层。
- 收益是否可量化（文件改动面、圈复杂度、缺陷率、交付速度）。
- 哪些改动应推迟到下一轮，避免过度重构。

研究完成后先形成内部摘要，再进入 Step 3 输出。

### Step 3: 输出结构化方案

按问题类型输出：

1. 业务驱动优化：
- 先给“业务理解确认”
- 再给“架构调整方案 + 迁移步骤 + 风险控制”
- 最后给“最小可实施版本（MVP Refactor）”

2. 技术驱动优化：
- 先给“技术债诊断清单”
- 再给“可直接执行的增量改造步骤”
- 包含 composable/store/service 拆分建议和样例骨架

3. 无代码架构设计：
- 输出“业务流程到架构映射图”
- 给出目录结构、模块边界、状态策略、接口层策略
- 说明未来功能扩展时如何保持稳定

## 表达与行为规范

- 先讲判断依据，再给建议。
- 不给“万能架构”，只给与当前业务阶段匹配的架构。
- 明确标出假设条件与不确定项。
- 建议默认可增量落地，不要求一次性重写。

## 反模式（必须避免）

- 未理解业务就重排目录和模块。
- 见重复就抽象，导致语义污染。
- 把状态管理分散在页面、组件、工具函数多处。
- 用“先进架构”替代“可维护架构”。
- 一次提交跨越过大，无法定位回归问题。

## 诚实边界

- 无法在缺少业务上下文时保证结构方案长期正确。
- 无法替代真实线上指标，只能提供架构层面的收益推断。
- 无法承诺一次重构解决所有历史问题，需分轮迭代。
- 对团队协作质量（代码评审、测试习惯）无法直接控制。

## 快速自检清单

- 是否先完成问题分类（业务驱动/技术驱动/无代码设计）？
- 业务驱动场景是否先确认业务规则再动结构？
- 是否给出可回滚的增量迁移步骤？
- 是否区分“可立即优化”与“需协同推进”的项？
- 是否说明风险、收益与适用边界？

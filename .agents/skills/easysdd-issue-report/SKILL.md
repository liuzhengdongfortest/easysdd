# easysdd-issue-report

本技能对应 easysdd-issue 工作流的**阶段一:问题报告**。

## 涉及的路径

> 共享路径与命名约定以根技能 `easysdd-core` 第二节和 `easysdd-issue` 第二节为准。本节只补充本阶段特有信息。

## 你的职责

引导用户把问题描述清楚,**不是替用户猜测问题原因**。问题报告里不写根因(那是阶段二的事),只写"发生了什么、怎么复现、应该怎样"。

## 启动检查

1. **确认是 BUG / 异常行为,而不是新功能需求**。如果用户描述的是"想加一个 X 功能",告诉他走 `easysdd-feature-design`。
2. **检查是否已有相关 issue 目录**。Glob `easysdd/issues/` 下的子目录,看有没有同类问题已经记录过;如果有,和用户确认是新建还是更新已有 report。
3. **快速通道判断**。这是 issue 工作流进入快速通道的**唯一正式判定点**。根据用户描述的线索先读一下代码(用 Grep/Read 定位相关文件),判断能否一眼确定根因:
   - **能**:根因明确(能给出 `{文件}:{行号}`),修复改动小(1-2 处),无跨模块影响风险 → 告诉用户"我已经看到问题所在,可以走快速通道:直接告知根因和修复方案,你确认后我立刻修,修完你验证,然后只写一份 fix-note.md"。等用户同意后触发 `easysdd-issue-fix`(快速通道模式)。
  - **不能**:根因有多个候选/不确定/需要更多复现信息 → 继续走下面的标准路径,做完整的问题报告。进入标准路径后,后续阶段默认不再二次改判快速通道。
4. **确定 issue 目录名**。和用户商定 slug,日期前缀用今天的日期(从环境信息里查 `currentDate`)。目录不存在就创建。(快速通道也需要建 issue 目录,fix-note.md 放在那里。)

## 必答 5 问

按顺序逐个问,**不要一次抛 5 个**。每个问题做一次模糊度检查,不通过继续追问。

### 1. 问题是什么?能描述一下你看到的现象吗?

- 期待:具体的异常表现。"点击提交按钮后,弹出了空白弹窗"比"提交功能有问题"有用一百倍。
- 模糊信号:"有时候会出错"、"感觉不对"——追问"具体什么时候"、"具体什么不对"。
- **红线:不要让用户描述根因**。出现"应该是因为 XXX"、"可能是 YYY 导致的"——记录现象,根因留给阶段二。

### 2. 怎么复现?

- 期待:一组最小复现步骤。例如:
  1. 进入 XX 页面
  2. 输入 YY 内容
  3. 点击 ZZ 按钮
  4. 看到问题现象
- 模糊信号:"不稳定复现"、"有时候能有时候不能"——追问复现频率和条件差异("什么情况下能复现")。如果确实不稳定复现,写明已知的触发条件和复现率。
- **"无法复现"也是有效回答**:写明"目前无法稳定复现,只在 X 情况下观察到一次"。

### 3. 期望行为 vs 实际行为

- 期待:两句话:
  - **期望**:我以为做了 A 之后应该发生 B
  - **实际**:但实际发生的是 C
- 不要合并成一句。期望和实际必须显式分开写,审查者才能快速判断问题边界。

### 4. 环境信息

- 最低要采集:**在哪个模块 / 功能区域发现的**、**相关文件或函数(如果用户知道)**
- 可选但有价值:操作系统、浏览器版本、运行环境(dev / prod)、最近有没有改过相关代码
- 如果用户说"我不知道在哪个文件"——没关系,写"待定",阶段二分析时会查

### 5. 严重程度与优先级

- 参考标准(给用户选):
  - **P0 - 阻塞**:核心功能完全失效,影响所有用户,必须立刻修
  - **P1 - 严重**:核心功能受损,有绕过方法,需要尽快修
  - **P2 - 中等**:非核心功能受损或影响少数用户,计划内修
  - **P3 - 轻微**:UI 瑕疵、边界情况、有更好的实现,按空闲修
- 如果用户不确定,帮他根据前几问的描述推荐一个,但让用户最终拍板

## 问题报告模板

让用户回答完后,把回答写进文件:

**路径**:问题报告(目录位置见主技能 `easysdd-issue` 第二节"目录安排")。

```markdown
---
doc_type: issue-report
issue: {issue 目录名}
status: draft
severity: P0 | P1 | P2 | P3
summary: {问题现象一句话}
tags: []
---

# {问题简述} Issue Report

## 1. 问题现象

{用户描述的具体异常表现,纯现象描述,不含根因推测}

## 2. 复现步骤

1. {步骤 1}
2. {步骤 2}
3. {步骤 3}
4. 观察到:{问题现象}

复现频率:{稳定复现 / 概率复现(约 X%) / 暂无法稳定复现}

## 3. 期望 vs 实际

**期望行为**:{做了 A 应该发生 B}

**实际行为**:{但实际发生了 C}

## 4. 环境信息

- 涉及模块/功能:{模块名或功能描述}
- 相关文件/函数:{已知文件:行号,或"待定"}
- 运行环境:{dev / staging / prod / 不确定}
- 其他上下文:{OS、浏览器、最近改动等,没有就写"无"}

## 5. 严重程度

**{P0 / P1 / P2 / P3}** — {一句话说明理由}

## 备注

{可选:用户额外补充的上下文、截图描述、日志片段等}
```

## 退出条件

写完文件后,跟用户做一次"出阶段一"确认:

- [ ] YAML frontmatter 存在，且 `doc_type=issue-report`、`issue` 与当前 issue 目录一致、`severity` 和 `summary` 非空、`tags` 至少 1 个
- [ ] 5 个问题都有具体答案,没有"待定"(环境中的相关文件可以是"待定")
- [ ] 问题现象是纯现象描述,没有混入根因推测
- [ ] 期望 vs 实际显式分开写
- [ ] 复现步骤可执行(哪怕是"暂无法稳定复现"也有说明)
- [ ] 用户明确说"report 可以了,进下一步"
- [ ] 用户确认后，frontmatter 的 `status` 已更新为 `confirmed`

## 退出后

告诉用户:"问题报告已就绪。下一步是阶段二:根因分析。可以触发 `easysdd-issue-analyze` 技能开始分析。"

不要自己顺手开始分析根因——阶段之间的人工 checkpoint 是工作流的硬约束。

## 反模式(看到就停)

- ❌ 用户说"可能是 XX 组件的问题",你顺着聊起来根因 —— 错,那是阶段二。现在只问"看到什么现象"
- ❌ 复现步骤太模糊("在用户界面操作一下")就放行 —— 错,逼出可执行的步骤
- ❌ 期望行为和实际行为混在一段话里 —— 错,必须显式分开
- ❌ 严重程度留空 —— 错,写"无" 或 给一个默认值,但不能空着
- ❌ 一口气把 5 个问题列成清单丢给用户填 —— 错,要逐题对话
